## The Azure Deployment Stack

Before touching application code, I needed somewhere to run it. The constitution mandates AKS with Cosmos DB serverless, Azure Key Vault with Managed Identity, and GitHub Actions with OIDC — no stored credentials.

Copilot generated the entire infrastructure-as-code layer in Bicep:

**Seven modules**, each responsible for exactly one Azure resource concern:

- **AKS** — Kubernetes 1.30, Azure Linux nodes, Calico network policy, workload identity enabled, Log Analytics integration. Two `Standard_B2s` nodes to start — the *Eurythmia* principle says nothing over-built.
- **ACR** — Basic SKU container registry. Admin user disabled. Images pushed via managed identity, not passwords.
- **Cosmos DB** — Serverless account with a `f1raceintelligence` database, `meetings` container partitioned by `/season`, `standings` container partitioned by `/season`. The partition strategy came directly from the data model spec: all reads are season-scoped.
- **Key Vault** — RBAC-authorized, soft-delete enabled with 7-day retention. No access policies — pure Azure RBAC.
- **Log Analytics** — 30-day retention workspace feeding AKS monitoring.
- **Workload Identity** — A user-assigned managed identity for the backend, federated to the AKS OIDC issuer. Granted AcrPull, Cosmos DB Data Contributor, and Key Vault Secrets User. The pod never sees a credential.
- **CI Identity** — A separate managed identity for GitHub Actions, federated to `https://token.actions.githubusercontent.com` for the `master` branch. Granted AcrPush and AKS Cluster User. The pipeline never sees a credential either.

Two bootstrap scripts tie it together: `deploy-infra.sh` provisions everything from an authenticated Azure CLI session, and `setup-github-secrets.sh` reads the deployment outputs and configures the nine GitHub Actions secrets automatically via `gh secret set`.

The CI/CD pipeline — which was a placeholder in Phase 1 — now has real jobs: OIDC login to Azure, ACR push with Docker Buildx, and Helm upgrade to AKS with image tag pinned to the commit SHA.

Region: `westus3`.

All from a conversation in the Copilot Chat panel.

## T005: The HTTP Server Bootstrap

The backend entry point. A Go HTTP server using Chi — a lightweight, stdlib-compatible router that adds just enough structure without pulling in a framework. The constitution's *Distributio* principle: nothing wasted.

Two files: `cmd/api/main.go` for the server lifecycle, `internal/api/router.go` for route registration. Two health endpoints from the start: `/healthz` for liveness and `/readyz` for readiness. Kubernetes needs both before the first real endpoint exists.

---

## T006: Cosmos DB Client and Repository Interfaces

This is the data layer that everything else depends on. Two interfaces — `CalendarRepository` and `StandingsRepository` — defined in `internal/storage/repository.go`. Pure Go interfaces. No Cosmos DB imports. No Azure SDK types leaking into the domain.

The concrete implementation lives in `internal/storage/cosmos/client.go`. It uses `DefaultAzureCredential` — which means it automatically picks up Managed Identity on AKS and falls through to Azure CLI credentials during local development. Parameterized queries throughout. Partition keys on season. Upsert semantics so polling is idempotent.

A `verify.go` file contains compile-time interface satisfaction checks:

```go
var (
    _ storage.CalendarRepository  = (*Client)(nil)
    _ storage.StandingsRepository = (*Client)(nil)
)
```

If the implementation drifts from the interface, the build breaks. Not a test. A compilation error. Immediate.

## T007: The OpenF1 Polling Scheduler

The first upstream integration. OpenF1 provides free, unauthenticated access to F1 meeting data — circuits, countries, dates, session schedules for all 24 rounds of 2026.

The poller runs on a 5-minute ticker (research decision #1: balance freshness against request churn). It fetches, normalizes into `RaceMeeting` domain structs, and upserts into Cosmos DB. If the upstream call fails, the last successful snapshot remains in the database with its `data_as_of_utc` timestamp — the frontend always has data, and always knows how fresh it is.

Standard library `time.Ticker`. Context cancellation for graceful shutdown. Structured `slog` logging. No cron library. No scheduler framework. *Distributio*.

---

## T008: The Hyprace Standings Client

Same pattern, different data source. Hyprace provides driver and constructor championship standings. The `HypraceClient` polls both endpoints on the same 5-minute cadence, normalizes into `DriverStandingRow` and `ConstructorStandingRow`, and upserts into the `standings` container.

Both pollers track their last successful poll time and last error — available for health checks and monitoring dashboards downstream.

## T009: The Frontend API Boundary

One file. Forty-five lines. But architecturally critical.

`frontend/src/services/apiClient.ts` is the *only* module the frontend uses to talk to the backend. Every component, every hook, every data fetch flows through this single gateway. The base URL comes from `VITE_API_BASE_URL` with a fallback to `/api/v1` — which means the Vite dev proxy and the production NGINX reverse proxy both work without code changes.

The constitution's *Decor* principle — propriety, fitness for purpose — says the frontend must never call upstream APIs directly. No OpenF1 calls from the browser. No Hyprace calls from the browser. The backend is the only data boundary. This file enforces that structurally.

---

## T010: Key Vault and Managed Identity

`internal/config/secrets.go` handles two things: environment-based configuration loading (`PORT`, `COSMOS_ACCOUNT_ENDPOINT`, `KEYVAULT_URI`) and Key Vault secret retrieval via `DefaultAzureCredential`.

No secrets in environment variables. No secrets in Helm values. No secrets in CI/CD logs. The pod authenticates to Key Vault via its federated workload identity, retrieves what it needs at startup, and proceeds. The *Decor* principle again: security infrastructure should look like security infrastructure.

---

## T011: Structured Logging and Metrics

Two files in `internal/observability/`:

**`logger.go`** — A JSON-structured `slog.Logger` writing to stdout. In AKS, stdout flows to the container runtime log, which the OMS agent ships to Log Analytics. No log aggregation library needed. The platform handles collection; the application handles structure.

**`metrics.go`** — Atomic counters for request count, request errors, poll successes, and poll failures. An HTTP middleware captures status codes and increments counters per request. A `Snapshot()` method returns the current values — ready for a `/metrics` endpoint or Azure Monitor integration later. Built on `sync/atomic`. No metrics framework. Nothing pulled in that isn't needed yet.

## T014: The Dependency Justification Ledger

The constitution mandates dependency discipline. Every third-party library must have an owner, a purpose, and a justification. The ledger lives at `specs/002-race-calendar-standings/dependency-justification.md`.

After Phase 2, the backend has exactly five dependencies beyond the standard library:

| Dependency | Purpose |
|---|---|
| `go-chi/chi` | HTTP router |
| `azure-sdk/azcore` | Azure SDK core |
| `azure-sdk/azidentity` | Azure credential |
| `azure-sdk/azcosmos` | Cosmos DB client |
| `azure-sdk/azsecrets` | Key Vault secrets |

The frontend has four: React, React DOM, Vite, TypeScript. All constitution-mandated.

Nine total dependencies across two services. The *Distributio* principle holds.

## What I Noticed in Phase 2

**The constitution is doing real work.** It is not a decorative document. When Copilot generated the Cosmos DB client, it used `DefaultAzureCredential` — not a connection string, not an access key. When it built the pollers, it used standard library tickers — not a cron framework. When it created the frontend API client, it enforced the tier boundary structurally. These decisions trace directly to constitution principles. The AI reads the rules and follows them.

**Interface-first design emerged naturally.** I didn't ask for the repository interfaces to be separate from the Cosmos implementation. The spec said "repository interfaces" and the AI produced clean Go interfaces in one package and the implementation in another. The compile-time checks were added to prove conformance. This is *Symmetria* — consistent standards applied without being told how.

**The deployment stack came together faster than the application code.** Seven Bicep modules, two Helm charts, two Dockerfiles, two bootstrap scripts, and a real CI/CD pipeline — all generated in one session. Infrastructure-as-code is highly structured and pattern-driven. It is exactly the kind of work AI agents excel at, because the patterns are well-established and the correctness criteria are clear.

**Dependency count matters.** Nine dependencies across two services is not an accident. It is a design decision. Every dependency is a maintenance commitment, a supply-chain risk, a future upgrade obligation. The ledger makes this visible. When someone proposes adding a library, the question is not "does it work?" but "does it justify its cost?" Vitruvius would recognize this immediately — *Distributio*, economy, nothing wasted.
